## Architecture â€” the content â†’ render pipeline

Content is **files, compiled at build**. There are four rendering paths; know which applies before editing a page:

| URL | Route | Content source | Renderer |
|---|---|---|---|
| `/blog`, `/blog/[slug]` | `app/blog/` | `content/blog/{en,es}/*.md` (frontmatter + markdown) | `lib/blog.ts` â†’ prose |
| `/methods/[slug]` | `app/methods/[slug]` | `content/pages/method/*.json` | `MethodTemplate` (structured) or `BlockRenderer` |
| `/specialties/[slug]` | `app/specialties/[slug]` | `content/pages/specialty/*.json` | `BlockRenderer` (blocks) |
| `/<slug>` (mission, team, faq, groups, insuranceâ€¦) | `app/[slug]` | `content/pages/bespoke/*.json` (blocks) **or** `*.md` (prose fallback) | `BlockRenderer` or `ContentPageView` |
| `/methods`, `/specialties` | overviews | `content/pages/overview/*.md` + card grid | prose + `PageCard` |

- **`lib/method-content.ts`** reads the `*.json` docs (`getMethod(slug, type)`, `getMethodCards`, `getMethodSlugs`) for `method | specialty | bespoke`. `MethodDoc` carries either structured fields (`hero/intro/features/cta`) **or** an ordered `blocks[]` array.
- **`lib/pages.ts`** reads the `*.md` prose docs (bespoke + overviews). A bespoke page prefers its `.json` blocks; falls back to `.md` prose when there are no blocks (careers, grief-and-loss, Spanish groups today).
- **`components/site/block-renderer.tsx`** â€” the faithful section renderer. `Block` union: `hero | text | image-text | cards | image | faq | table | cta`. This is how Divi layouts are reproduced 1:1 (2-col hero, feature-card grids, image+text, accordions, tables). Adding a Divi layout pattern = add a block type here.
- **`components/site/method-template.tsx`** â€” the fixed hero+intro+feature-grid+cta template used by most method pages (approved look). Has a prose fallback (`bodyHtml`) for methods whose Divi layout doesn't fit.

### How content JSON/MD is generated (offline scripts, in the scratchpad â€” NOT part of the app)

Content docs were produced by Python scripts in the session scratchpad (`parse_blocks.py` = rendered-Divi-HTML â†’ typed blocks; `parse_divi.py` = Divi shortcodes â†’ structured method fields; `blog_to_md.py`, `batch_*.py`). To re-derive a page: fetch the **original** page HTML (the untouched live pages, or Wayback, or `_recovery/og_site/*.html` when provided), run the parser, map images, write the JSON. See the resume notes file (gitignored) for exact locations. **Do not hand-edit generated JSON as the primary workflow** â€” fix the parser and re-run, or edit the JSON for one-off tweaks.

## Conventions & gotchas

- **Nav** is defined in `lib/nav.ts` (mirrors the original menu exactly â€” keep it faithful). Site identity/contact in `lib/site.ts`.
- **URLs preserve the original**: methods `/methods/<slug>`, specialties `/specialties/<slug>`, bespoke top-level `/<slug>`. Old dated blog permalinks (`/YYYY/MM/DD/slug/`) â†’ **301 via `next.config.ts` `redirects()`** (from generated `lib/legacy-redirects.ts`); ~1,042 hacked-spam URLs â†’ **410 via `next.config.ts` `rewrites()` â†’ `app/api/gone`**. **NOT `proxy.ts`** â€” a Next-16 proxy is Node-runtime and OpenNext/Cloudflare hard-rejects Node middleware (empirically verified). See `docs/seo/REDIRECTS.md`.
- **Breadcrumbs**: none, anywhere (removed site-wide 2026-07-17 at the user's request â€” `components/site/breadcrumb.tsx` was deleted; don't reintroduce).
- **Images**: all local under `public/images/` (`library/` = page images, `blog/`, `brand/`, `home/`). Many original photos are **Divi composites with a green overlay panel** â€” crop that panel off with `sharp` before use (helper approach in the resume notes). In **dev**, the first render of a `next/image` may show blank (on-demand optimization latency) â€” warm it (`curl /_next/image?url=â€¦`) or scroll; it's fine in the static build. The dev `next/image` cache is also **sticky per URL** â€” if you re-crop a file under the same name it keeps serving the stale crop through cache clears/restarts; use a new filename (prod `next build` is unaffected).
- **Team cards** (`/team`): each member's branded green name-plate is drawn in CSS by `components/site/team-photo.tsx` (`<TeamPhoto>`), not baked into the image â€” triggered by a `card: {name,title,languages,objectPosition}` on an `image-text` block. Raw headshots live in `source-images/team/` (outside `public/`, not deployed); `sharp`-optimized cards go in `public/images/team/cards/`. Bios use the Read-More toggle (`expandable-bio.tsx`, `motion`) via `more-text-content` markers. Full add-a-member flow: `source-images/team/README.md`.
- Markdown â†’ HTML via `lib/markdown.ts` (`unified` + `remark`/`rehype`, sanitized). Divi page bodies also render sanitized HTML.
- The live WordPress site is **actively compromised** (malware currently deactivated, not removed). Never import/execute anything from `_recovery/` backups â€” they are evidence. See `docs/INCIDENT-FINDINGS.md`.

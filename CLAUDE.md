## ⚠️ This is NOT the Next.js you know (from AGENTS.md)

This project runs **Next.js 16.2 / React 19.2 / Tailwind v4**, which have breaking changes vs. training-data knowledge. **Read the relevant guide in `node_modules/next/dist/docs/` before writing Next code.** Verified breakages already hit in this repo:
- `middleware.ts` → **`proxy.ts`** (`export function proxy`, Node runtime only).
- Page/layout `params` and `searchParams` are **Promises** — always `await`. Type route props with the global `PageProps<'/route'>` (generated by `next typegen`; no import).
- Tailwind v4 is **CSS-first**: no `tailwind.config.js`. Config lives in `app/globals.css` (`@theme`, `@plugin`, `@import "tailwindcss"`). A `webpack` key in `next.config.ts` fails the build (Turbopack is default).
- React 19: `useActionState` (from `react`), not `useFormState`.
Full corrections table: `docs/MIGRATION-PLAN.md`.

## What this is

A rebuild of **growthjourneytherapy.com** — a bilingual (EN/ES) therapy practice — migrating off a **hacked WordPress/Divi** site to Next.js + local markdown (no CMS, no database, no auth). All content was **recovered** from the compromised site (`_recovery/`, see `docs/POST-RECOVERY.md`, `docs/INCIDENT-FINDINGS.md`). Design must stay **1:1 with the original Divi pages**.

## Architecture — the content → render pipeline

Content is **files, compiled at build**. There are four rendering paths; know which applies before editing a page:

| URL | Route | Content source | Renderer |
|---|---|---|---|
| `/blog`, `/blog/[slug]` | `app/blog/` | `content/blog/{en,es}/*.md` (frontmatter + markdown) | `lib/blog.ts` → prose |
| `/methods/[slug]` | `app/methods/[slug]` | `content/pages/method/*.json` | `MethodTemplate` (structured) or `BlockRenderer` |
| `/specialties/[slug]` | `app/specialties/[slug]` | `content/pages/specialty/*.json` | `BlockRenderer` (blocks) |
| `/<slug>` (mission, team, faq, groups, insurance…) | `app/[slug]` | `content/pages/bespoke/*.json` (blocks) **or** `*.md` (prose fallback) | `BlockRenderer` or `ContentPageView` |
| `/methods`, `/specialties` | overviews | `content/pages/overview/*.md` + card grid | prose + `PageCard` |

- **`lib/method-content.ts`** reads the `*.json` docs (`getMethod(slug, type)`, `getMethodCards`, `getMethodSlugs`) for `method | specialty | bespoke`. `MethodDoc` carries either structured fields (`hero/intro/features/cta`) **or** an ordered `blocks[]` array.
- **`lib/pages.ts`** reads the `*.md` prose docs (bespoke + overviews). A bespoke page prefers its `.json` blocks; falls back to `.md` prose when there are no blocks (careers, grief-and-loss, Spanish groups today).
- **`components/site/block-renderer.tsx`** — the faithful section renderer. `Block` union: `hero | text | image-text | cards | image | faq | table | cta`. This is how Divi layouts are reproduced 1:1 (2-col hero, feature-card grids, image+text, accordions, tables). Adding a Divi layout pattern = add a block type here.
- **`components/site/method-template.tsx`** — the fixed hero+intro+feature-grid+cta template used by most method pages (approved look). Has a prose fallback (`bodyHtml`) for methods whose Divi layout doesn't fit.

### How content JSON/MD is generated (offline scripts, in the scratchpad — NOT part of the app)

Content docs were produced by Python scripts in the session scratchpad (`parse_blocks.py` = rendered-Divi-HTML → typed blocks; `parse_divi.py` = Divi shortcodes → structured method fields; `blog_to_md.py`, `batch_*.py`). To re-derive a page: fetch the **original** page HTML (the untouched live pages, or Wayback, or `_recovery/og_site/*.html` when provided), run the parser, map images, write the JSON. See the resume notes file (gitignored) for exact locations. **Do not hand-edit generated JSON as the primary workflow** — fix the parser and re-run, or edit the JSON for one-off tweaks.

## Design system (`app/globals.css`)

- **Brand tokens** (from the pre-hack Divi CSS): `--green #2f7863` (primary), `--sage #a1b75b`, gold/terracotta accents, warm neutrals. Mapped to shadcn semantic tokens; brand utilities like `bg-green`, `text-ink`, `bg-sage`, `text-on-green`. Fonts: **Poppins** (display/headings) + **Lato** (body) via `next/font`.
- **Accessible type scale** (primary user is 65+/low-vision; see `docs/ACCESSIBILITY-TYPE-SCALE.md`): root `html { font-size: 118.75% }` → **19px body** (GOV.UK/NHS standard). All `--text-*` tokens overridden; **never use `text-[13px]`-style hardcoded px** (defeats zoom/resize) — use the scale, 16px floor. Reading columns capped with `max-w-prose`.
- Container width `--container-max: 1500px` (matches original Divi 1501px).
- `Button` (`components/ui/button.tsx`) variants: `solid | outline | outline-light | ghost`, sizes `sm|md|lg`. Use it, not raw `<a>`, so target sizes stay ≥44px.

## Conventions & gotchas

- **Nav** is defined in `lib/nav.ts` (mirrors the original menu exactly — keep it faithful). Site identity/contact in `lib/site.ts`.
- **URLs preserve the original**: methods `/methods/<slug>`, specialties `/specialties/<slug>`, bespoke top-level `/<slug>`. Old dated blog permalinks (`/YYYY/MM/DD/slug/`) are legacy → will 301/410 via `proxy.ts` (SEO plan in `docs/seo/`, not yet built).
- **Breadcrumbs**: none, anywhere (removed site-wide 2026-07-17 at the user's request — `components/site/breadcrumb.tsx` was deleted; don't reintroduce).
- **Images**: all local under `public/images/` (`library/` = page images, `blog/`, `brand/`, `home/`). Many original photos are **Divi composites with a green overlay panel** — crop that panel off with `sharp` before use (helper approach in the resume notes). In **dev**, the first render of a `next/image` may show blank (on-demand optimization latency) — warm it (`curl /_next/image?url=…`) or scroll; it's fine in the static build. The dev `next/image` cache is also **sticky per URL** — if you re-crop a file under the same name it keeps serving the stale crop through cache clears/restarts; use a new filename (prod `next build` is unaffected).
- **Team cards** (`/team`): each member's branded green name-plate is drawn in CSS by `components/site/team-photo.tsx` (`<TeamPhoto>`), not baked into the image — triggered by a `card: {name,title,languages,objectPosition}` on an `image-text` block. Raw headshots live in `source-images/team/` (outside `public/`, not deployed); `sharp`-optimized cards go in `public/images/team/cards/`. Bios use the Read-More toggle (`expandable-bio.tsx`, `motion`) via `more-text-content` markers. Full add-a-member flow: `source-images/team/README.md`.
- Markdown → HTML via `lib/markdown.ts` (`unified` + `remark`/`rehype`, sanitized). Divi page bodies also render sanitized HTML.
- The live WordPress site is **actively compromised** (malware currently deactivated, not removed). Never import/execute anything from `_recovery/` backups — they are evidence. See `docs/INCIDENT-FINDINGS.md`.

## 0. THE CONTRADICTION LIST ‚Äî where local docs differ from training-data assumptions

This is the most important section. Every item was verified in the local docs.

1. **`middleware.ts` is DEPRECATED. It is now `proxy.ts` with `export function proxy(request)`.**
   - `DOCS/03-api-reference/03-file-conventions/proxy.md`: "The `middleware` file convention is
     deprecated and has been renamed to `proxy`."
   - `DOCS/02-guides/upgrading/version-16.md` ("`middleware` to `proxy`"): "The `edge` runtime is
     **NOT** supported in `proxy`. The `proxy` runtime is `nodejs`, and it cannot be configured."
   - Config flags renamed too: `skipMiddlewareUrlNormalize` ‚Üí `skipProxyUrlNormalize`.
   - Do NOT write `middleware.ts`. Write `proxy.ts` at repo root.

2. **`params` and `searchParams` are Promises, and synchronous access is FULLY REMOVED (not just deprecated).**
   - `version-16.md` "Async Request APIs (Breaking change)": "Starting with **Next.js 16**,
     synchronous access is fully removed. These APIs can only be accessed asynchronously." Applies to
     `cookies`, `headers`, `draftMode`, `params` in layout/page/route/default/opengraph-image/icon,
     and `searchParams` in page.

3. **New globally-available typed helpers `PageProps<'/route'>` and `LayoutProps<'/route'>`** ‚Äî
   generated by `next dev` / `next build` / `next typegen`, no import needed.
   - `DOCS/03-api-reference/03-file-conventions/page.md` "Page Props Helper".
   - Prefer these over hand-written `{ params: Promise<...> }` types.

4. **`opengraph-image.tsx` / `icon.tsx` image functions now receive `params` AND `id` as Promises**
   (`generateImageMetadata` still gets sync params). `version-16.md` "Async parameters for icon, and
   open-graph Image (Breaking change)".

5. **`sitemap.ts` with `generateSitemaps`: `id` is now `Promise<string>`** (was a number!).
   `version-16.md`: "the `sitemap` generating function now receives `id` as a promise" and the
   example does `Number(await id)`. (We won't need `generateSitemaps` ‚Äî one sitemap is plenty ‚Äî but
   don't assume the old signature.)

6. **Turbopack is the default for BOTH `next dev` and `next build`.** Any `webpack` config makes the
   build FAIL. Custom loaders go under top-level `turbopack` key (no longer `experimental.turbopack`).
   `version-16.md` "Turbopack by default". Consequence: any markdown pipeline must be plain
   JS at runtime/build-time (unified/remark), or a Turbopack-compatible loader ‚Äî another reason to
   avoid webpack-era MDX loader tricks.

7. **PPR flag is gone; caching is opt-in via `cacheComponents: true`** and the `'use cache'` /
   `cacheLife` / `cacheTag` directives (now stable, no `unstable_` prefix). `revalidateTag` now takes
   a REQUIRED second `cacheLife` profile argument (`revalidateTag('posts', 'max')`); new `updateTag`
   and `refresh` APIs exist. `version-16.md` "Caching APIs". (Mostly irrelevant to a fully static
   site, but do not write `revalidateTag('x')` one-arg calls.)

8. **`next/image` breaking changes** (`version-16.md` "`next/image` changes"):
   - `images.qualities` now defaults to `[75]` only; other `quality` props get coerced.
   - `images.minimumCacheTTL` default 60s ‚Üí 4 hours.
   - `16` removed from default `imageSizes`.
   - Local `src` with a query string now requires `images.localPatterns.search`.
   - `images.domains` deprecated (use `remotePatterns`) and `next/legacy/image` deprecated.
   - New: image optimization of local IPs blocked by default; `images.maximumRedirects` default 3.

9. **Images without static imports: there is now an OFFICIAL dynamic-`import()` pattern.**
   `DOCS/01-getting-started/12-images.md` "Images without static imports": "you can use a dynamic
   `import()` in a Server Component to still get automatic `width`, `height`, and `blurDataURL`"
   ‚Äî `const { default: image } = await import(`../content/blog/images/${imageFilename}`)`, then
   `<Image src={image} />` with NO width/height props. "The path must include a static prefix ‚Ä¶
   **all** files matching that prefix are bundled." This replaces the old "you must hardcode
   width/height for /public paths" reflex for content images.

10. **The `i18n` key in `next.config` is Pages-Router-only ‚Äî it does not exist for App Router.**
    There is no `i18n.md` in `DOCS/03-api-reference/05-config/01-next-config-js/` (verified by `ls`);
    the App Router guide `DOCS/02-guides/internationalization.md` prescribes `proxy.js` negotiation +
    an `app/[lang]` segment + dictionaries as the DIY path, or `next-intl`/`next-international`.

11. **`next dev`/`next build` no longer need `--turbopack`; `next lint` is removed** (codemod
    migrates to the ESLint CLI ‚Äî this repo's `package.json` already has `"lint": "eslint"`).
    Node.js ‚â• 20.9 and TypeScript ‚â• 5.1 required. `version-16.md` top section.

12. **`generateStaticParams` + ISR nuance:** "You must return an empty array from
    `generateStaticParams` or utilize `export const dynamic = 'force-static'` in order to revalidate
    (ISR) paths at runtime" ‚Äî `DOCS/03-api-reference/04-functions/generate-static-params.md`.
    For us: return the FULL list and set `export const dynamicParams = false` so unknown slugs 404.

13. **Proxy runs on Node.js runtime ‚Äî so it CAN read the filesystem / use Node APIs** (no edge
    restrictions), but docs warn it "is meant to be invoked separately of your render code ‚Ä¶
    you should not attempt relying on shared modules or globals" (`proxy.md`).

14. **Metadata can be streamed**; `generateMetadata` resolution "is part of rendering the page"
    (`DOCS/03-api-reference/04-functions/generate-metadata.md`). Also unchanged-but-worth-noting:
    file-based metadata (opengraph-image.png etc.) has HIGHER priority than the `metadata` object.

## 3. Markdown strategy (Q3) ‚Äî RECOMMENDATION: parse ourselves (gray-matter + unified), NOT @next/mdx

What the docs support (`DOCS/02-guides/mdx.md`):
- Official path is `@next/mdx` + `@mdx-js/loader` + mandatory root `mdx-components.tsx`
  ("**required** to use `@next/mdx` with App Router and will not work without it").
- `.md` handling needs `extension: /\.(md|mdx)$/` opt-in.
- **"`@next/mdx` does not support frontmatter by default"** (line 622) ‚Äî it suggests
  remark-frontmatter / gray-matter yourself.

Why self-parsing wins for THIS site:
- Content is prose written by a therapist: no embedded React components needed, so MDX's only
  advantage is dead weight (and an MDX compile step per file under Turbopack).
- We need a **content index** anyway (frontmatter: title, lang, date, pair-slug, category, legacy
  WordPress URLs) to drive: blog listing pages, `generateStaticParams`, hreflang `alternates`,
  `sitemap.ts`, and the legacy-redirect map. That means reading files with `fs` + `gray-matter`
  regardless ‚Äî at which point rendering the body with `remark-parse ‚Üí remark-rehype ‚Üí rehype-react`
  (or `-stringify`) in the same pass is simpler than maintaining a parallel MDX pipeline.
- File-based MDX routing (`page.mdx` per post) can't express the bilingual pairing or per-post
  metadata cleanly, and puts content inside `app/` where the lead agent's routing lives.
- Typography: Tailwind v4 + `@tailwindcss/typography` `prose` classes on the rendered article.
  i18n: `lang` lives in frontmatter; the renderer is language-agnostic. Code highlighting: not needed.

Pipeline (all Turbopack-safe, pure JS, runs in Server Components at build time):
`gray-matter` ‚Üí `unified().use(remarkParse).use(remarkGfm).use(remarkRehype).use(rehypeSlug)`
‚Üí custom image handling (below) ‚Üí `rehype-react` (render to React elements so `<img>` nodes become
our `<PostImage>` server component) ‚Äî or `rehype-stringify` + `dangerouslySetInnerHTML` if we accept
plain `<img>` (don't; see Q5).

## 4. Bilingual routing (Q4) ‚Äî RECOMMENDATION: option (ii), route groups `(en)` / `(es)` with literal localized slugs

Docs basis:
- `DOCS/02-guides/internationalization.md` recommends proxy-based `Accept-Language` negotiation +
  an `app/[lang]/‚Ä¶` segment. **That pattern forces a locale prefix on every URL** (`/en-US/products`)
  ‚Äî which would destroy every one of her existing URLs (`/contacto/`, `/category/psicologia/`).
  Since preserving exact old Spanish root-level slugs is a hard SEO requirement, option (i) is out
  unless we add a rewrite layer to hide the prefix (needless complexity, and locale-detection
  redirects are actively bad for SEO on a bilingual public site ‚Äî Google crawls without cookies).
- `DOCS/03-api-reference/03-file-conventions/route-groups.md`: route groups are exactly for
  "Defining multiple root layouts"; caveats: cross-group navigation between different root layouts is
  a full page load (fine ‚Äî EN and ES sections are near-disjoint), and "Routes in different groups
  should not resolve to the same URL path" (our EN/ES slugs differ by construction: `/contact` vs
  `/contacto`, `/category/psychology` vs `/category/psicologia`).
- Middleware negotiation (iii) alone can't give localized slugs; it only picks a locale. Rejected as
  the primary mechanism; we don't auto-redirect by language at all (both languages are first-class,
  discoverable via hreflang + visible language switcher).

Two root layouts give us correct `<html lang="en">` vs `<html lang="es">` per section with zero
runtime machinery. Blog posts: EN posts at `/blog/[slug]` (in `(en)`), ES posts at `/es/blog/[slug]`
(in `(es)`), because two groups may NOT both claim `/blog/[slug]`. Old dated WP permalinks
(`/2023/05/01/que-es-la-deprivacion-cultural/`) get 301'd to the new location either way, so the ES
blog prefix costs nothing; her root-level Spanish PAGE slugs are preserved exactly.

## 5. Blog images from markdown (Q5)

`DOCS/01-getting-started/12-images.md`:
- Statically imported images: width/height/blurDataURL automatic.
- `/public`-path string src: **still requires manual `width` and `height`** ("you'll need to provide
  the width, height ‚Ä¶ manually" applies whenever Next can't introspect the file at build time; the
  local-image example with `src="/profile.png"` passes explicit `width={500} height={500}`).
- **The right pattern for markdown-referenced images** ‚Äî "Images without static imports":

  > "If you can't use a static `import` for your images, you can use a dynamic `import()` in a
  > Server Component to still get automatic `width`, `height`, and `blurDataURL`"

  ```tsx
  // components/post-image.tsx (Server Component)
  import Image from 'next/image'
  export async function PostImage({ file, alt }: { file: string; alt: string }) {
    const { default: image } = await import(`@/content/blog/images/${file}`)
    return <Image src={image} alt={alt} sizes="(max-width: 768px) 100vw, 720px" />
  }
  ```

  Doc requires "a static prefix (like `../content/blog/images/`)" ‚Äî all matching files are bundled;
  path aliases (`@/`) explicitly supported.

Wire-up: in the unified pipeline, map markdown `![alt](file.jpg)` nodes to `<PostImage>` via
`rehype-react` component mapping. So markdown authors write plain relative filenames; rendering
gets optimized `next/image` with automatic dimensions and blur placeholders ‚Äî zero CLS, no manual
width/height bookkeeping. Remember the v16 gotchas: quality defaults to 75-only, no query strings
on local srcs without `localPatterns.search`.

## 6. Metadata / SEO shape (Q6)

Per `DOCS/03-api-reference/04-functions/generate-metadata.md`:
- Set `metadataBase: new URL('https://growthjourneytherapy.com')` once in each root layout
  ("typically set in root `app/layout.js`"); then `alternates.canonical` and `alternates.languages`
  take relative paths. Head output is `<link rel="canonical">` + `<link rel="alternate" hreflang=‚Ä¶>`
  exactly as shown in the doc's `<head> output` block (lines ~416-418).
- EN/ES pairs: EVERY paired page declares BOTH languages in `alternates.languages` (and each page's
  canonical is itself), e.g. contact:

```tsx
// app/(en)/contact/page.tsx
export const metadata: Metadata = {
  title: 'Contact',
  alternates: { canonical: '/contact', languages: { en: '/contact', es: '/contacto' } },
}
// app/(es)/contacto/page.tsx mirrors it with canonical: '/contacto', same languages map.
```

- `title.template` (`generate-metadata.md` ¬ßtemplate): set in root layouts, applies to children only.
- OG images, two mechanisms (`DOCS/03-api-reference/03-file-conventions/01-metadata/opengraph-image.md`):
  - Static file convention: `opengraph-image.(jpg|jpeg|png|gif)` + `opengraph-image.alt.txt` in any
    route segment; auto-generates og:image/width/height/type tags. File-based beats the metadata object.
  - Code: `opengraph-image.tsx` default-exporting an `ImageResponse` ‚Äî **in v16 its `params` (and
    `id`) props are Promises** (`version-16.md`). Use code-gen for per-post OG cards (title text on
    brand background), static file at `app/opengraph-image.png` as the site default.

## 7. ~1,000 legacy redirects (Q7)

Reality check first: only ~30-80 of those URLs are HER content (post ids 1222-2004 + ~20 pages);
the other ~965 are Russian pharma spam (ids ‚â•234527) that must NOT be redirected ‚Äî return **410 Gone**
so Google deindexes them fast.

What this version offers (`DOCS/02-guides/redirecting.md`):
- `async redirects()` in `next.config.ts` ‚Äî supports path matching + regex. Doc: "`redirects` may
  have a limit on platforms. For example, on Vercel, there's a limit of 1,024 redirects. To manage a
  large number of redirects (1000+), consider creating a custom solution using Proxy." Also:
  "`redirects` runs **before** Proxy."
- Proxy (`NextResponse.redirect`, any status code) + a JSON redirect map; for very large maps the doc
  recommends a Bloom filter + Route Handler lookup ("Managing redirects at scale (advanced)").

Recommended layered design (small, fast, no config bloat):
1. **`content/redirects/legacy.json`** generated from Wayback + frontmatter `legacyUrls`
   (~60-100 entries: dated permalinks, `/faqs`‚Üí`/faq`, `?p=<id>` forms, category feeds).
   `next.config.ts` imports it: `async redirects() { return legacy.map(...) }` ‚Äî a data file, not
   config bloat; well under any platform limit; compiled into the routes manifest, matched before
   anything renders. Add ONE regex rule for `?p=` ids if we map them.
2. **`proxy.ts` for spam suppression**: pattern-match known-spam shapes (percent-encoded Cyrillic in
   path, `/2026/` dated urls, `/category/uncategorized/‚Ä¶`) and return a bare 410
   (`new Response('Gone', { status: 410 })`). Proxy is Node runtime in v16, no edge limits; matcher
   scoped so it never runs on `/_next/*` or real routes.
3. Everything else falls through to the app; unknown blog slugs 404 via `dynamicParams = false` + `not-found.tsx`.

Perf tradeoff: config `redirects()` is O(n) linear scan per request against the manifest ‚Äî trivial at
n‚âà100, and it short-circuits before proxy and rendering. Pushing thousands of entries into it is what
the docs warn about; we avoid that by 410ing spam in proxy (one regex test) instead of enumerating it.

## 8. Sitemap + robots (Q8)

File conventions in `app/` root (`DOCS/03-api-reference/03-file-conventions/01-metadata/sitemap.md`, `robots.md`):

```ts
// app/sitemap.ts ‚Äî "special Route Handler that is cached by default"
import type { MetadataRoute } from 'next'
import { getAllPosts, getAllPages } from '@/lib/content'
const BASE = 'https://growthjourneytherapy.com'

export default function sitemap(): MetadataRoute.Sitemap {
  return [
    ...getAllPages().map((p) => ({ url: `${BASE}${p.path}`, lastModified: p.updated })),
    ...getAllPosts('en').map((p) => ({ url: `${BASE}/blog/${p.slug}`, lastModified: p.date })),
    ...getAllPosts('es').map((p) => ({ url: `${BASE}/es/blog/${p.slug}`, lastModified: p.date })),
  ]
}
```

```ts
// app/robots.ts
import type { MetadataRoute } from 'next'
export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: '*', allow: '/' },
    sitemap: 'https://growthjourneytherapy.com/sitemap.xml',
  }
}
```

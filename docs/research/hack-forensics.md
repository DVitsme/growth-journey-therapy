# Hack Forensics Runbook — growthjourneytherapy.com
Date: 2026-07-09. Scope: identify the hack family, and define the safe procedure to mine the
compromised DB dump for the owner's legitimate content. Nothing here touches the repo.

---

## 1. Fingerprint: what hack is this?

**Verdict: automated SEO doorway-spam post injection (Russian/Cyrillic variant of the
"Japanese keyword hack" family), performed through an authenticated WordPress channel —
NOT WP-VCD, NOT Balada, NOT SocGholish.**

Evidence match:

| Observation | What it tells us |
|---|---|
| 1,047 posts, 965 dumped into category 1 "Uncategorized" | Posts inserted via `wp_insert_post()`/REST/XML-RPC with no category set → WP assigns the default category. This is the signature of programmatic insertion, not manual posting. |
| Cyrillic slugs ("как-онлайн-среда-влияет...") | Russian-language doorway spam targeting Yandex/Google.ru — the Russian analog of the Japanese keyword hack Google documents (autogenerated pages + sitemap abuse). |
| Post IDs jump from ~2004 (her real posts, per Wayback `?p=` URLs) and ~231303 (her real pages) to 234527+ | Autoincrement gap = tens of thousands of rows inserted **and deleted** over time, or ID-jumping by bulk INSERT. Confirms long-running automated campaign. |
| New post today (2026-07-09 16:09) | Live persistence: a wp-cron task, a backdoor plugin/mu-plugin, or an external bot with working credentials/application password. The attacker still has write access RIGHT NOW. |
| Yoast v27.4 live, `/post-sitemap.xml` returns 200 | Spam is being auto-sitemapped — the whole point of this family: get doorways indexed fast. Sucuri's June 2025 "Hidden Spam Pages" writeup describes exactly this: spam posts "in the tens or sometimes even hundreds of thousands," injected after admin access was obtained, concealed from wp-admin with helper plugins. |

Why not the other candidates:
- **Japanese keyword hack (classic)**: same *family* and monetization, but classically drops static
  `.html` files in random directories; here the spam lives in `wp_posts`. Treat this as the
  Russian-language, database-resident variant of that family.
- **WP-VCD**: comes from nulled themes/plugins; injects spam *links* + rogue admin `100010010`;
  does not mass-create 1,000 Cyrillic posts. (Still check for its files — see §2.)
- **Balada Injector**: JS redirect injections via plugin vulns (tagDiv etc.), rogue admins;
  payload is script injection, not post creation.
- **SocGholish**: fake-browser-update JS delivery; no post spam. Not this.
- **2024–2026 plugin-abuse wave** (e.g. WP-Automatic CVE-2024-27956, unauth SQLi → admin users +
  spam content): *plausible as the ENTRY vector* — these campaigns end in exactly this state
  (rogue admin, mass spam posts, backdoored plugins).

**Likely entry vector on Bluehost shared hosting**, in order of probability:
1. Vulnerable/outdated plugin or theme (Elementor ecosystem add-ons are a top 2023–2026 target)
   giving privilege escalation or arbitrary post creation.
2. Compromised admin password or **Application Password** abused via `/wp-json` or `xmlrpc.php`
   (brute force on wp-login/xmlrpc is endemic; the REST API is exactly how you insert 1,000
   categorized-as-default posts).
3. **Cross-site contamination**: any other addon domain in the same Bluehost cPanel account is in
   the same filesystem/user; one infected neighbor infects all (Sucuri lists this as a leading
   reinfection cause on shared hosting).
4. Stolen cPanel/FTP credentials (info-stealer malware on a PC that had saved credentials).

Onset date: unknown until we have the dump. First command after extraction:
`SELECT MIN(post_date) FROM wp_posts WHERE ID > 100000;` and
`SELECT MIN(post_date) FROM wp_posts WHERE post_title REGEXP '[А-Яа-я]';`
Everything after that timestamp is presumed hostile.

## 2. Where this family persists — the distrust list

Assume ALL of these are dirty in BOTH backups until proven otherwise:

1. **wp_posts** — the spam itself (post_type `post`, maybe `page`; check ALL post_types).
2. **wp_options** —
   - injected autoloaded options (`autoload='yes'`) containing base64/serialized payloads;
   - the **`cron` option**: malicious scheduled hooks are how "new post every day" happens with
     no attacker connection. Grep the cron blob for hook names that aren't core/known-plugin.
   - concealment options (Sucuri saw `hide_pages_posts_options` hiding spam IDs from wp-admin);
   - poisoned `siteurl`/`home`, `default_role`, `users_can_register`, `active_plugins`
     (fake plugin slugs), `template`/`stylesheet`.
3. **wp_users** — rogue admins. Also check for changed email on HER account. WP-VCD uses id
   100010010; others use names like `wp-admin`, `adminbackup`, `wordpress`, random hex.
4. **wp_usermeta** — `wp_capabilities` escalated to administrator for a subscriber; rogue
   `session_tokens`; hidden-admin tricks (meta that filters the user out of the Users list).
5. **`wp-content/mu-plugins/`** — must-use plugins run on every request with no activation and
   no listing in the Plugins screen. Anything here is suspect by default.
6. **PHP anywhere under `wp-content/uploads/`** — 100% malicious, no exceptions. Also
   double-extension files (`image.php.jpg`, `x.jpg.php`), `.ico`/`.png` files that are actually
   PHP (check magic bytes), and `.htaccess` files inside uploads enabling PHP execution.
7. **`.htaccess`** (root and nested) — conditional redirects keyed on referer/user-agent
   (Google-only cloaking), `AddHandler`/`SetHandler` tricks, rewrite to doorway scripts.
8. **Prepended/appended code** in `wp-config.php`, active theme `functions.php`, root
   `index.php`, `wp-settings.php`, `wp-load.php` — `eval(base64_decode(...))`,
   `gzinflate(str_rot13(...))`, `include` of a file in uploads, `@include "\x2f..."` hex paths.
9. **Fake plugin directories** mimicking real names (`yoast-seo-links`, `performance-labs`,
   `wp-zzz`, `pluginn`, one-letter dirs) — Sucuri's 2025 case used a fake "Yoast SEO Links"
   plugin as a passwordless-login backdoor plus a "Performance Labs" plugin that hid the spam
   posts from admins via CSS. These often fatal-error on deactivation attempts.
10. **wp_postmeta** — payloads parked in meta (`_wp_attached_file` path tricks, serialized
    blobs, Yoast fields stuffed with spam), and cloaking flags marking doorway posts.
11. **wp_termmeta / wp_terms** — spam tag clouds, thousands of Cyrillic tags.
12. **Attacker Search Console ownership** — this family adds itself as a verified owner
    (via uploaded HTML token or meta tag) to submit spam sitemaps. Check for
    `google[a-f0-9]+.html` files and `wp_options` verification meta. After rebuild, audit
    Search Console users & delete stale verification tokens.
13. **Outside WP entirely** (cPanel layer): rogue cron jobs in cPanel, extra FTP accounts,
    SSH keys in `~/.ssh/authorized_keys`, files in the account but outside `public_html`,
    other addon-domain docroots. The .wpress backup won't even show you these; the cPanel
    backup will — inspect its `cron/`, `sshkeys/`, `homedir/` listings.

Corollary: **never restore either backup into a running WordPress/PHP environment.** Treat both
archives as evidence bags. Extract only two things: the SQL dump and `wp-content/uploads/`
image files. Nothing else crosses the boundary.

## 3. THE CENTRAL QUESTION — extracting her real posts from wp_posts

### Ground rules
- **Whitelist, never blacklist.** She has roughly 10–80 real posts (1,047 total − 965 spam-bucket
  ≈ 82 upper bound, minus revisions/attachments/nav items). At that scale you hand-review every
  survivor. Filtering out spam is fragile; filtering IN known-good is safe.
- **Parse the dump offline.** Best: a throwaway Dockerized MariaDB with no WordPress attached
  (`docker run --rm -e MARIADB_ROOT_PASSWORD=x -v scratch:/work mariadb:11`), or a pure-Python
  INSERT parser. Before importing, grep the dump for `TRIGGER|PROCEDURE|INTO OUTFILE|LOAD_FILE`
  and strip anything found — a SQL dump can't execute PHP, but don't import routines blindly.
- Mind the table prefix: it may not be `wp_` — read it from `wp-config.php` inside the backup
  ( `$table_prefix` ) or from the dump's CREATE TABLE lines.

### Filter ranking

| Filter | Reliability | Notes |
|---|---|---|
| **ID range** (`ID <= ~2100` for posts) | **Best single generic filter.** Wayback shows her original posts at `?p=1222`–`?p=2004`; spam starts ≥234527. Autoincrement never reuses IDs, so no spam can exist below her ceiling — *unless the attacker edited/replaced low-ID rows in place* (see §6). Her 2025-era PAGES live at 230964–231303; handle pages as a second explicit ID window. |
| **Wayback slug whitelist** (`post_name IN (...)`) | Gold standard for *selection* — it's externally verifiable ground truth the attacker can't rewrite. Build the list from the 1,486-URL CDX inventory. Weakness: misses never-archived drafts/posts. |
| `post_date <` onset | Strong, but attacker controls `post_date` and CAN backdate. Use as corroboration, not primary. |
| `post_type='post' AND post_status='publish'` | Necessary noise reduction, not a spam filter. |
| `post_author` | Useful once you've confirmed her user ID(s) — but attackers frequently post AS the compromised admin, so it cannot exonerate a row. |
| Category (term_relationships → term_taxonomy 1) | 965/1047 spam is in cat 1, but "in Uncategorized" ≠ spam and "in psychology" ≠ clean. Corroboration only. |
| Language/charset (Cyrillic detection) | Great tripwire — she writes EN + ES only. Any `[Ѐ-ӿ]` in title/slug/content of a "clean" row = red flag. |

### Belt-and-braces extraction query

```sql
-- 1) candidate set: whitelist by ID window + type/status
SELECT ID, post_author, post_date, post_modified, post_status,
       post_name, post_title, post_content, post_excerpt
FROM wp_posts
WHERE post_type = 'post'
  AND post_status IN ('publish','draft')       -- keep drafts for review
  AND ID BETWEEN 1 AND 2100                     -- her original post window
ORDER BY ID;

-- 2) tripwires on the candidate set (any hit => quarantine that row for manual diff vs Wayback)
--   a. Cyrillic anywhere:        post_title/post_content/post_name REGEXP '[а-яА-ЯёЁ]'
--   b. modified after onset:     post_modified >= '<ONSET_DATE>'
--   c. external anchors:         post_content LIKE '%<a %href="http%' to non-whitelisted domains
--   d. script/iframe/base64:     post_content REGEXP '<script|<iframe|base64,|eval\\('
--   e. author not in her known user IDs

-- 3) cross-check: every slug from the Wayback CDX post list must appear in the result.
--    Missing slug => the row was deleted/retyped/trashed → go to §6 (revisions).
```

Repeat with `post_type='page' AND ID BETWEEN 230900 AND 231400` (plus any older page IDs found
under 2100) for the ~20 known pages, and `post_type='attachment'` only to resolve featured
images (take the file *path*, then fetch the actual file from uploads/, never trust `guid`).

**Then a human reads every surviving post.** At ≤80 rows this takes an hour and is the only
step that catches semantic tampering (a paragraph of casino links appended to a real 2023 post).

## 4. Sanitizing post_content on the way to Markdown

Threat model: her *real* posts may carry injected `<script>`, hidden keyword-stuffed divs,
spam anchors, base64 payloads. **The concrete risk is stored XSS: if the rebuild ever renders
recovered HTML via `dangerouslySetInnerHTML` (or you preview a recovered .html file in your own
browser), injected script executes.** Also: never open extracted HTML files directly in a
browser during triage — read them in an editor, or `python3 -m http.server` is NOT safe either;
editor only.

Concrete pipeline (Python, offline):

```
raw wp_posts.post_content
  → 1. unwrap WP block comments (<!-- wp:paragraph --> etc.) and strip ALL HTML comments
  → 2. strip shortcodes: whitelist [caption] (keep inner img+text), drop every other [...]
  → 3. sanitize with nh3 (Rust/ammonia bindings; bleach is deprecated but acceptable):
       allowed tags:  p a img h1-h6 ul ol li blockquote strong em b i br hr
                      figure figcaption table thead tbody tr td th code pre
       allowed attrs: a[href] img[src,alt] td/th[colspan,rowspan]
       url schemes:   https, http, mailto  → data: URIs are DROPPED
       (nh3 removes script/iframe/object/embed/form/style, all on* event handlers,
        javascript: URLs, and all style attributes by default — that also kills
        display:none keyword-stuffing divs' cloak while keeping their text VISIBLE,
        so step 4 must catch them by content, not by style)
  → 4. link & language audit (before styles are gone, run on RAW too):
       - drop every element whose style matched /display\s*:\s*none|visibility\s*:\s*hidden|
         text-indent\s*:\s*-\d{3,}|position\s*:\s*absolute;.*(left|top)\s*:\s*-\d{3,}/ 
         (run this on the RAW html PRE-sanitize, since nh3 strips style attrs)
       - every <a href> to a domain NOT in {growthjourneytherapy.com, well-known refs she
         actually cites}: unwrap to plain text (keep anchor text, delete href) and log it
       - reject/flag any remaining base64 blob, any Cyrillic run in an EN/ES post
  → 5. rewrite internal links & images:
       - <img src> must resolve to her uploads/ tree; download from the extracted uploads
         (or Wayback im_/) into /public/images/blog/, re-encode with sharp/imagemagick
         (re-encoding strips polyglot payloads & EXIF), verify magic bytes are image types
       - internal hrefs → new site routes
  → 6. convert to Markdown (markdownify or pandoc -f html -t gfm-raw_html).
       CRITICAL: emit with raw HTML disabled — everything the sanitizer allowed maps to pure
       Markdown; anything that can't map gets dropped, not passed through.
  → 7. diff each result against the Wayback id_ capture of the same post (text-level diff).
       Unexplained additions = tampering caught.
```

In the Next.js app, render these .md files with a Markdown renderer that does **not** allow raw
HTML passthrough. Never `dangerouslySetInnerHTML` recovered content, sanitized or not.

## 5. wp_postmeta — keep almost nothing

Keep ONLY these keys, ONLY for whitelisted post IDs, treated as **plain text**:
- `_yoast_wpseo_title`, `_yoast_wpseo_metadesc`, `_yoast_wpseo_focuskw`
- `_thumbnail_id` → resolve to the attachment row → take `_wp_attached_file` (a relative path),
  then pull that file from uploads and verify it's a real image.

Validation per value: reject if it contains `<`, `>`, `base64`, a URL to a foreign domain,
Cyrillic (for EN/ES posts), or is implausibly long (a meta description over ~300 chars is
poisoned or junk). Strip to plain text regardless — Yoast meta should never contain markup.

Discard everything else. Specifically NEVER carry:
- any serialized PHP value (`a:{...}`, `O:{...}`) — and never run `unserialize()` on dump data
  in a PHP context (PHP object injection);
- `_elementor_data` / `_elementor_*` — huge untrusted JSON; if you need page copy from it,
  extract text nodes with a JSON parser, never render it;
- `_wp_page_template`, `_edit_lock`, `_edit_last`, oembed caches, `_menu_item_*`, anything
  from unknown plugins.

## 6. Were her ORIGINAL posts deleted or overwritten? (assume maybe — verify)

Precedent: the 2017 WP REST API campaign *replaced* existing posts' content with spam
(Sucuri documented legitimate content being removed, not just added to). So yes, real risk.

Detection, in order:

1. **Slug reconciliation (external truth).** Set A = post slugs from the Wayback CDX inventory.
   Set B = `post_name` values in the dump with `post_type='post'`. `A − B` = deleted posts.
   For each, recover content from Wayback raw capture (`/web/<ts>id_/<url>`).
2. **ID-gap scan.** `SELECT ID FROM wp_posts WHERE ID BETWEEN 1 AND 2100 ORDER BY ID;` — gaps are
   normal (revisions/autosaves get IDs too), but a gap where a Wayback `?p=` URL exists
   (1222, 1225, 1227, 1948–2004) = a deleted real post.
3. **Status check.** `... WHERE ID <= 2100 AND post_status IN ('trash','draft','auto-draft','inherit','private')`
   — trashed rows still hold content (`post_name` gets a `__trashed` suffix; original slug is in
   postmeta `_wp_trash_meta_status`/`_wp_desired_post_slug`). Recover and review them.
4. **Tamper check.** `post_modified` far later than `post_date` on a low-ID post = edited after
   the fact. Diff against Wayback.
5. **Mine revisions — the DB's own time machine.**
   ```sql
   SELECT r.ID, r.post_parent, r.post_date, r.post_name, r.post_title,
          LENGTH(r.post_content) AS len
   FROM wp_posts r
   WHERE r.post_type = 'revision'
     AND r.post_parent BETWEEN 1 AND 2100
   ORDER BY r.post_parent, r.post_date;
   ```
   How revisions work: every save copies title/content/excerpt into a new row with
   `post_type='revision'`, `post_status='inherit'`, `post_parent=<original ID>`,
   `post_name='<parent>-revision-v1'` (autosaves: `<parent>-autosave-v1`). Even when a live row
   was overwritten — or trashed — its revision rows usually survive, because deleting a post
   only cascades if done through WP's `wp_delete_post`, and spam campaigns rarely bother.
   **Pitfall:** if the attacker tampered via the editor/REST, the tampered save created a NEW
   revision too. So per parent, walk revisions chronologically and take the newest revision
   whose `post_date` precedes hack onset AND whose content passes the §4 tripwires; diff it
   against the live row to see exactly what changed. Revisions of a deleted parent are matched
   purely by `post_parent` = the missing ID.
6. Note: `wp_posts.guid` for old posts encodes the original `?p=` URL
   (`https://site.com/?p=1222`) — a handy secondary key for matching Wayback `@p=` captures to
   dump rows. Never publish or trust GUIDs as URLs, though.

## 7. Definitive DO-NOT-CARRY-FORWARD list for the Next.js rebuild

Carry forward ONLY: (a) sanitized Markdown of whitelisted, human-reviewed posts & page copy;
(b) plain-text Yoast title/description/focus keyword for those posts; (c) re-encoded image
files from uploads/ (magic-byte-verified, EXIF-stripped, renamed); (d) her category names
(psychology / psicologia) and the URL slugs (for redirects/SEO continuity).

NEVER carry forward:
1. Any PHP file, from anywhere. The target stack has no PHP; if PHP "needs" to come along,
   something is wrong.
2. Themes, plugins, mu-plugins, Elementor templates/JSON, `_elementor_data`.
3. `.htaccess` rules, nginx snippets, `wp-config.php` constants, salts, DB creds, API keys —
   rotate every secret that ever lived on that server (Resend gets brand-new keys).
4. wp_users / wp_usermeta — no accounts exist in the new site at all (no auth by design).
5. wp_options — nothing from it. Not widgets, not the cron blob, not "theme mods".
6. wp_comments / wp_commentmeta — comment spam farm; new site has no comments.
7. Raw post_content HTML rendered as HTML (`dangerouslySetInnerHTML`) — Markdown only.
8. Any file from uploads that is not a verified raster image she actually used: no .php, .js,
   .html, .ico, .svg (SVG = XSS vector; if a logo must be SVG, rebuild it or sanitize with
   svgo + manual review), no double extensions, no files with image extensions but non-image
   magic bytes.
9. The XML sitemaps, robots.txt, or any Yoast redirect exports from the compromised site —
   generate fresh ones in Next.js from the whitelisted content.
10. Any hotlinked/external image URL, tracking pixel, or third-party embed found in old content.
11. Spam posts, obviously — but also: do NOT 301 the ~965 spam URLs anywhere. Serve 410 Gone
    for the Cyrillic-slug pattern so Google drops them fast, and 301 only the whitelisted real
    slugs to their new URLs.
12. Google Search Console: after launch, audit owners/users, remove unknown verifications,
    remove attacker-submitted sitemaps, and use URL Removal for residual spam indexing.
13. Anything from the OLD hosting account into the new deploy pipeline: no SSH keys, no cron
    entries, no FTP accounts, no reused passwords.

## Appendix A — selective extraction under the 4.6 GB disk limit

Both backups are 9.7 GB zips; full extraction is impossible. Pull ONLY the SQL dump and uploads.

**Backup B (cPanel full backup = zip containing backup-*.tar.gz):**
```bash
S=/tmp/claude-1000/-home-nero-Clients-growthjourneytherapy/9b67c8f1-0efe-424a-863e-5378b9be898a/scratchpad
unzip -l backupB.zip                                   # find the inner tar.gz name
unzip -p backupB.zip 'backup-*.tar.gz' | tar -tzf - > "$S/cpanel-manifest.txt"   # list only, no extraction
grep -Ei 'mysql/.*\.sql|/uploads/' "$S/cpanel-manifest.txt" | head -50
# cPanel keeps DB dumps under <backup>/mysql/<dbname>.sql — extract just that, streamed:
unzip -p backupB.zip 'backup-*.tar.gz' | tar -xzf - -C "$S" --wildcards '*/mysql/*.sql'
# uploads, images only, streamed (repeat per year dir if space is tight):
unzip -p backupB.zip 'backup-*.tar.gz' | tar -xzf - -C "$S" --wildcards \
  '*/public_html/wp-content/uploads/*' \
  --exclude='*.php*' --exclude='*.js' --exclude='*.html' --exclude='*.ico' --exclude='*.svg'
```
Also grep the manifest for gold: `homedir/.ssh/`, `cron/`, `mu-plugins/`, `uploads/*.php` —
each hit is forensic evidence of persistence (look, don't extract).

**Backup A (.wpress, possibly zip-wrapped):** All-in-One WP Migration's .wpress is a custom
sequential format: per file, a 4,377-byte header block (255B name, 14B size, 12B mtime, 4096B
path), then the raw bytes; EOF = 4,377 zero bytes. No index, so you must stream it — use a
small Python reader that walks headers and writes ONLY `database.sql` (it's usually near the
end, so the walk is I/O-bound but disk-free) and optionally uploads images. Open-source
extractors exist (search github "wpress-extract"); vet whichever you use, or write the ~40-line
reader from the format above. If the .wpress is inside a zip, stream with
`unzip -p backupA.zip '*.wpress' | python3 wpress_stream.py --only database.sql`.

**Dump pre-import hygiene:**
```bash
grep -Ein 'TRIGGER|PROCEDURE|INTO OUTFILE|LOAD_FILE|DEFINER' "$S"/*.sql | head   # expect empty
grep -m1 'CREATE TABLE `[a-z0-9_]*posts`' "$S"/*.sql                             # learn real prefix
docker run -d --name forensic-db -e MARIADB_ROOT_PASSWORD=x -v "$S":/work mariadb:11
docker exec forensic-db sh -c 'mariadb -px -e "CREATE DATABASE f" && mariadb -px f < /work/<dump>.sql'
```
Then run the §3 queries. Compare the TWO dumps against each other — the backups are
"compromised in different ways," so `diff` of their whitelisted-row exports is itself a
tamper detector: any row that differs between backups was edited between backup dates.
